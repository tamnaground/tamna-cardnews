# /harvest — 파이프라인 수확

세컨드 브레인의 지식을 한 단계씩 앞으로 흘려보낸다 (inbox → projects → output → wiki).
루트 `CLAUDE.md`의 흐름 규칙을 따른다. 아래 순서대로 수행하라.

## 1단계 — inbox 분류

`inbox/`의 노트를 하나씩 읽고 (\_index.md 제외) 다음 중 하나로 처리한다:

- **기존 프로젝트 소재** → 해당 `projects/` 파일의 적절한 섹션에 내용을 병합하고 inbox 노트 삭제.
- **새 작업 씨앗** (사용자가 착수 의사를 밝힌 것만) → `projects/`에 새 파일 생성(`status: active`, `created` frontmatter) 후 inbox 노트 삭제. 착수 여부가 불분명하면 만들지 말고 inbox에 둔다.
- **아직 판단 불가** → 그대로 둔다. inbox에 남는 것은 실패가 아니다.
- **명백한 쓰레기/중복** → 삭제 후보로 보고만 하고, 사용자 확인 전에는 삭제하지 않는다.

inbox 노트를 wiki로 직접 보내지 않는다. 어떤 경우에도 inbox 노트를 "다듬어서" 다시 inbox에 두지 않는다.

## 2단계 — 완료 프로젝트 수확

`projects/`에서 `status: done`인 파일을 찾는다. 각각에 대해:

1. **발행물 확인**: 산출물이 `output/YYYY-MM-DD_이름/` 폴더에 `발행노트.md`와 함께 있는지 확인. 없으면 지금 만든다 (발행노트에는 발행일·채널·산출물 목록·제작 방법 요약).
2. **wiki 증류**: 프로젝트에서 **시간이 지나도 참인 것**만 골라 wiki에 반영한다 — 검증된 표현·규칙·교훈·레퍼런스. 기존 wiki 문서에 병합 가능하면 병합하고, 새 문서가 필요할 때만 만든다. 모든 항목에 출처(output 발행노트 링크)를 남긴다. 일회성 세부사항(특정 마감, 임시 결정)은 버린다.
3. **실행형 노하우 분리**: 도구 실행 절차·버그 회피법은 wiki가 아니라 `.claude/skills/`의 해당 스킬에 반영한다.
4. **프로젝트 파일 삭제**: 내용이 output/wiki/skills로 모두 수확되었으면 프로젝트 파일을 삭제한다. 애매하면 삭제하지 말고 보고한다.

`status: active | paused` 프로젝트는 건드리지 않는다.

## 3단계 — 현황판 갱신

`inbox/_index.md`, `projects/_index.md`, `output/_index.md`, `wiki/_index.md`를 실제 파일 상태와 일치하게 갱신한다. 깨진 상대 링크가 없는지 확인한다.

## 4단계 — 보고와 커밋

- 무엇을 어디로 옮겼는지, 무엇을 wiki에 증류했는지, 삭제 후보는 무엇인지 요약해 보고한다.
- 변경 사항을 한국어 메시지로 커밋한다 (예: `수확: 실사영상 프로젝트 완료 처리, 제주어-표현 갱신`).
