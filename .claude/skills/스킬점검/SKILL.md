# 스킬 점검 (Skills Audit)

커스텀 스킬을 전수 조사해서 keep / merge / rewrite / delete 판정을 내리고, 가장 나쁜 것부터 직접 고친다.

## 0단계 — 대상 수집

세 위치를 전부 확인한다. 하나만 보고 끝내지 말 것:

```bash
find ~/.claude/skills "$PWD/.claude/skills" -name "SKILL.md" 2>/dev/null
ls ~/.claude/plugins 2>/dev/null   # 플러그인 제공 스킬
```

**엣지 케이스:**
- 하니스 내장 스킬(세션 스킬 목록에는 보이지만 디스크에 파일이 없는 것)은 점검 대상이 아니다. 사용자에게 "이건 내장이라 수정 불가"라고 명시할 것
- 사용자가 말한 경로에 스킬이 거의 없으면, 다른 위치(프로젝트 `.claude/skills/` 등)도 찾아서 실제 점검 가능한 전체 목록을 먼저 보고할 것
- frontmatter의 `name:`과 디렉터리명이 다르면 그 자체가 결함이다

## 1단계 — 스킬별 3축 평가

각 SKILL.md를 전부 읽고(설명만 보지 말 것) 아래 3축으로 평가한다:

**① 트리거 정확도 (description 품질)**
- 오발동: 설명이 너무 넓어서 무관한 요청에 발동하는가? ("~세팅", "~작업" 같은 포괄어 주의)
- 미발동: 실제 사용 상황의 표현이 빠졌는가? 사용자는 기능명이 아니라 **증상**("테스트가 안 돌아", "대사 바꿔줘")으로 말한다 — 증상 표현이 트리거에 있는지 확인
- 신규 제작만 트리거하고 수정·재실행·검증 요청을 놓치지 않는가?
- 경계(부정 조건)가 있는가? "X는 이 스킬 말고 Y로" 한 줄이 오발동을 크게 줄인다

**② 겹침/충돌**
- 다른 커스텀 스킬·내장 스킬·MCP 서버와 담당 영역이 겹치면, 어느 쪽을 쓸지 경계가 본문이나 설명에 명시되어 있는가?
- 겹치는데 내용이 거의 같으면 merge 후보

**③ 실질 가치**
- 이 스킬이 없으면 매번 다시 알아내야 하는 정보(실제 겪은 버그, 경로, 명령어, 브랜드 규칙)가 담겨 있는가? → 가치 높음
- 일반 상식이나 모델이 이미 아는 내용만 있으면 → 죽은 스킬(delete 후보)
- 본문 내부 모순(지시와 템플릿이 충돌), 하드코딩된 보고 문구, 실행 불가능한 명령이 있으면 rewrite 후보

## 2단계 — 판정 표

결과를 표로 정리한다: | 스킬 | 트리거 정확도 | 겹침/충돌 | 가치 | 판정 |

판정 기준:
- **keep**: 3축 모두 양호. 설명만 다듬는 것은 keep에 포함
- **merge**: 다른 스킬과 내용 중복이 절반 이상
- **rewrite**: 가치는 있으나 설명 또는 본문에 구조적 결함(모순, 애매한 지시, 엣지 케이스 누락)
- **delete**: 가치 없음, 또는 내장 기능과 완전 중복

## 3단계 — 직접 수정

판정만 하고 끝내지 말고 최악부터 고친다:

1. **설명 재작성**: 가장 나쁜 설명들을 다시 쓴다. 좋은 설명의 형식 = "무엇을 하는 스킬인지 한 문장" + "발동 조건 나열(증상 표현 포함)" + "부정 조건(안 쓸 때)". 존재하는 스킬 수가 요청된 개수보다 적으면 있는 만큼 전부 고치고 그 사실을 보고한다
2. **본문 수정**: 가장 문제 많은 SKILL.md 하나를 고른다. 우선순위: 자기모순 제거 > 애매한 표현("적절히", "필요하면")을 판정 가능한 조건으로 교체 > 실제 놓친 엣지 케이스 추가. 길이를 늘리는 것이 목적이 아니다 — 지시가 판정 가능해지는 것이 목적

## 4단계 — 반영과 보존

- 프로젝트 `.claude/skills/` 파일은 지정 브랜치에 커밋·푸시
- `~/.claude/skills/` 파일은 원격 컨테이너에서는 세션 종료 시 사라진다 — 수정했으면 반드시 사용자에게 알리고, 레포로 옮길지 물을 것 (로컬 세션이면 해당 없음)
- 최종 보고에는 판정 표, 수정한 항목별 before/after 요지, 커밋 해시를 포함한다
