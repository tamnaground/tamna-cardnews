# 탐나그라운드 세컨드 브레인

Claude Code가 관리하는 Obsidian 호환 지식 저장소.
지식을 주제별 폴더에 쌓지 않고, **작업 단계를 따라 흐르게** 한다 (Karpathy 스타일 파이프라인).

```
수집          작업            발행           증류
inbox/  ──▶  projects/  ──▶  output/  ──▶  wiki/
빠르게 던짐    프로젝트당       배포된         완성 후에만
정리 금지      파일 1개        결과물만        정제해서 축적
```

## 폴더 규칙

### inbox/ — 수집

- 떠오른 생각, 링크, 아이디어, 조사거리를 **마찰 없이** 던져 넣는 곳.
- 파일 하나 = 생각 하나. 제목은 대충 붙여도 된다. 정리·분류·태깅 금지.
- inbox 노트가 지저분한 것은 정상이다. Claude는 inbox 노트를 다듬지 않는다.
- 분류는 오직 `/harvest` 실행 시에만 한다.

### projects/ — 진행 중인 작업

- **프로젝트당 파일 1개.** 조사 메모, 대사, 초안, 할 일, 작업 로그를 전부 그 한 파일 안에 누적한다.
- frontmatter에 `status: active | paused | done`을 유지한다.
- 결과물이 배포되면 `status: done`으로 바꾼다 → 다음 `/harvest`의 수확 대상이 된다.

### output/ — 발행물

- 실제로 배포·발행된 결과물(이미지, 영상, 코드, 글)만 둔다.
- 발행물 하나 = `YYYY-MM-DD_이름/` 폴더 하나. 폴더 안에 산출물 파일과 `발행노트.md`(무엇을, 어디에, 어떻게 만들었는지)를 둔다.
- **output은 불변이다.** 여기 파일은 수정하지 않는다. 후속 작업이 필요하면 새 프로젝트를 만든다.

### wiki/ — 증류된 지식

- **무언가 실제로 완성된 뒤에만** 쓴다. 처음부터 wiki 문서를 직접 작성하지 않는다.
- 시간이 지나도 참인 것만 남긴다: 검증된 표현, 재사용 가능한 규칙, 얻은 교훈, 정리된 레퍼런스.
- 모든 항목에 어느 output/프로젝트에서 나왔는지 출처를 남긴다.
- 갱신은 `/harvest`를 통해서만 한다.

## 흐름 규칙

- 지식은 **앞으로만** 흐른다: inbox → projects → output → wiki.
- 단계를 건너뛰지 않는다. 특히 inbox → wiki 직행 금지.
- wiki 내용이 새 프로젝트에 필요하면 복사하지 말고 링크로 참조한다.
- 각 폴더의 `_index.md`는 그 폴더의 현황판이다. 파일을 만들거나 옮기면 반드시 함께 갱신한다.

## 작성 규칙

- **순수 마크다운만.** Obsidian 플러그인 문법(Dataview, templater 등) 금지.
- 링크는 상대 경로 마크다운 링크로: [제주어 표현](wiki/제주어-표현.md) 처럼 (projects 파일에서라면 `../wiki/제주어-표현.md`). 위키링크 `[[...]]`도 쓰지 않는다.
- 파일명: 한글 허용, 공백 대신 하이픈(`-`), 발행물 폴더만 날짜 접두사 `YYYY-MM-DD_`.
- frontmatter는 projects 파일에만 필수(`status`, `created`). 나머지는 선택.
- 문서 첫 줄은 `# 제목`, 그 아래에 한 줄 요약.

## Claude 운영 규칙

- 사용자가 자료·아이디어를 던져주면 → `inbox/`에 노트로 저장하고 끝. 그 자리에서 정리하지 않는다.
- 새 작업을 시작하면 → `projects/`에 파일 1개 생성 (`status: active`).
- 결과물이 배포되면 → `output/YYYY-MM-DD_이름/` 폴더 생성, 산출물 + `발행노트.md` 저장, 프로젝트를 `status: done`으로.
- `/harvest` 실행 시에만 wiki를 갱신하고 done 프로젝트를 정리한다.
- 실행형 도구 노하우(영상 렌더링 파이프라인 등)는 wiki가 아니라 `.claude/skills/`에 둔다. wiki에는 링크만 남긴다.
- 커밋은 의미 있는 단위로, 한국어 메시지로 남긴다.
